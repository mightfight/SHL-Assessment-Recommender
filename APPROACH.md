# SHL Assessment Recommender – Approach Overview

This document outlines the design and implementation of the web‑based RAG recommendation tool built against the SHL product catalogue.  The goal is to provide context, highlight key components, and show how the system meets the evaluation criteria.

---

## 1. Data Acquisition & Storage

1. **Scraping pipeline**
   * `backend/scraper.py` implements a robust crawler for SHL’s public product catalog.
   * Listings are paginated; links to individual assessments are extracted with BeautifulSoup.
   * Detail pages are parsed using text‑line heuristics to reliably capture name, description, duration, test type, job levels, languages, and support flags.
   * Scraped records are stored as JSON in `backend/data/assessments.json`.

2. **Vector store creation**
   * `backend/embeddings.py` reads the JSON and constructs a rich text representation for each assessment.
   * Gemini embedding API (`models/gemini-embedding-001`) generates vectors with batching and rate‑limit handling.
   * ChromaDB persistent collection is used as the retrieval index; embeddings and metadata are stored locally (`backend/data/chroma_db`).
   * A first‑run initializer in `backend/app.py` auto‑builds the store if empty.

> **Retrieval mechanism:** semantic similarity search over Gemini embeddings in ChromaDB.

---

## 2. RAG Recommendation Engine

1. **Retrieval**
   * `recommender.search()` queries the vector store using a new Gemini embedding of the user input.
   * Top‑20 nearest neighbours (cosine) are returned as candidate assessments.

2. **Query handling**
   * Free text, job description URLs (fetched with `requests` + BeautifulSoup), or short queries are accepted.
   * URL inputs are converted to plain text (truncated to 3 000 chars) before embedding.

3. **Re‑ranking with LLM**
   * When a valid `GEMINI_API_KEY` is configured, `recommender.rerank_with_gemini()` prompts the `gemini-2.5-flash` model.
   * The LLM is instructed to choose 5–10 assessments per query, balancing technical (K) vs. behavioral (P) tests and obeying explicit rules derived from requirements.
   * The prompt is engineered to return a strict JSON object containing selected indices; fallback to similarity ranking if the call fails or exceeds quota.

4. **API & Frontend**
   * `backend/app.py` exposes a FastAPI service with `/health` and `/recommend` (POST & GET) endpoints.
   * Request/response models are defined with Pydantic.
   * Frontend static files (`frontend/`) provide a polished single‑page UI that talks to `/recommend` and shows results in a table.
   * The app can be launched locally via `python run.py` (default host `0.0.0.0:8000`).

5. **Outcome structure**
   * Recommendations returned as JSON with fields: `url`, `name`, `adaptive_support`, `description`, `duration`, `remote_support`, `test_type`.

---

## 3. Evaluation Methodology

* `evaluation/evaluate.py` processes the labelled training set contained in `Gen_AI Dataset.xlsx`.
* Recall@10 is computed per query, normalizing URLs to a canonical form.
* Mean Recall@10 is reported along with a detailed per‑query log saved to `evaluation/eval_results.json`.
* A simple script (`evaluation/generate_predictions.py`) runs the pipeline over the provided test queries and writes results to `Firstname_Lastname.csv` (submission format).

> **Metrics at key stages:**
> * Scraper: number of assessments scraped plus coverage of description/type/duration.
> * Embeddings: count of vectors in ChromaDB.
> * Retrieval: candidate count per query.
> * Final evaluation: Recall@K on the labelled dataset.

---

## 4. Deployment & URLs

* **Local server:** `http://localhost:8000` serves the frontend and API.
* **API endpoint:** POST `http://<host>:8000/recommend` with JSON `{"query":"..."}` returns recommendations in JSON.
* The webapp can be containerised or hosted on any platform supporting Python 3.11–3.12 and the stated requirements.

### Conclusion
This solution satisfies the requirements by:
1. Scraping, parsing, and storing the SHL product catalogue with quality checks.
2. Employing a modern RAG architecture—ChromaDB + Gemini embeddings plus LLM re‑ranking—for query understanding and recommendations.
3. Providing evaluation scripts and reporting metrics to measure accuracy.

Further enhancements could include caching, more sophisticated prompt tactics, or alternative embedding/LLM providers, but the core pipeline is complete and functional.
